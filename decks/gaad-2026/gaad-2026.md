### What is an AI-Native Design System? {data-slide-tag-name=rh-slide slot=heading}

A design system whose component APIs, design tokens, and metadata are
structured so that AI agents can **generate correct, accessible UI**
while reducing hallucinations.

Not a new design system. A new way to **describe** one.

### CEM: Component Manifest {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="ui" icon="code"></rh-icon>
Generates a **component manifest** from source code.

One command: `cem generate`

A JSON schema describing every attribute, slot, CSS property,
event, method, demo, and guideline.

Standards-based:
[Custom Elements Manifest](https://github.com/webcomponents/custom-elements-manifest) spec.

---

### cem health {data-slide-tag-name=rh-slide slot=heading}

Documentation quality scoring. 6 dimensions, 100 points:

- Element description (25 pts) -- **a11y notes**, **keyboard interaction**
- Attribute docs (20 pts) -- purpose, constraints
- Slot docs (15 pts) -- **content guidelines**, **a11y considerations**
- CSS docs (15 pts) -- properties, parts
- Event docs (15 pts) -- triggers, detail shape
- Demos (10 pts)

Good docs serve **cognitive a11y** for humans
AND **richer context** for AI.

---

### Asimonim: Design Tokens {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="ui" icon="ai-search"></rh-icon>
Design tokens parser, validator, and language server.

*"What is the brand red?"*
&rarr; `--rh-color-brand-red-on-light: #ee0000`

Multi-format export: CSS, SCSS, TypeScript, Swift, Android XML.

Standards-based:
[DTCG](https://design-tokens.github.io/community-group/format/) spec.

---

### MCP: A USB Port for AI {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="standard" icon="model-context-protocol"></rh-icon>
**MCP** (Model Context Protocol) is like a USB port for AI.

Plug in your design system. AI can use it.

**CEM MCP** + **Asimonim MCP** + **Figma MCP** -- all in one session.

- `generate_html` -- AI generates valid component markup
- `validate_html` -- AI checks its own work
- `search_tokens` -- AI queries real design token values
- `cem://accessibility` -- WCAG patterns and ARIA guidance

---

### Language Server {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="standard" icon="command-line"></rh-icon>
`cem lsp` -- IDE support that goes where your templates are:

- HTML, TypeScript template literals
- **Twig**, Jinja2, Blade, PHP, Nunjucks
- Handlebars, Liquid, ERB, EJS

Autocomplete, hover docs, validation with autofixes,
go-to-definition.

Meeting template authors where they are.

---

### Mappa: Performance Is Accessibility {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="ui" icon="speedometer"></rh-icon>
High-performance **import map** generator. Go + tree-sitter.

~72x faster than alternatives. Powers CEM's dev server.

Unbundled ES modules: smaller payloads, faster loads,
better experience on constrained devices and networks.

Standards-based:
[Import Maps](https://html.spec.whatwg.org/multipage/webappapis.html#import-maps) spec.

---

### Architecture {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="standard" icon="architecture"></rh-icon>

- **CEM** uses **Asimonim** to enrich component docs with token values
- **CEM** uses **Mappa** for dev server import map generation
- All three share duck-typed FileSystem interfaces

| Layer      | Tool     | Standard                 |
| ---------- | -------- | ------------------------ |
| Components | CEM      | Custom Elements Manifest |
| Tokens     | Asimonim | DTCG                     |
| Modules    | Mappa    | Import Maps              |
