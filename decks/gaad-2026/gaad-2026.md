## Custom elements manifest

A **web standard format** for describing web component APIs.
Attributes, slots, events, CSS properties, demos, and guidelines
-- in one JSON file.

Like `package.json` for your component library.
Used by IDEs, documentation tools, and AI agents to understand how components work.

---

### Standards-based development {data-slide-tag-name=rh-slide slot=heading}

## The `cem` CLI

`cem` is our open-source tool for working with custom elements manifests.
It's a high-performance, cross-platform go binary which provides support for the
entire <abbr>ASDL</abbr>, from idea to PR review.

- `cem generate` -- extract a manifest from RHDS source code
- `cem health` -- score documentation quality
- `cem serve` -- interactive component dev server
- `cem lsp` -- IDE autocomplete and validation
- `cem mcp` -- AI native access to design system elements

---

### High performance tooling {data-slide-tag-name=rh-slide slot=heading}

## The `cem health` command

Documentation quality scoring. Good docs serve **cognitive a11y** for humans
AND **richer context** for AI. We analyze documentation and score
across six dimensions, 100 points:

- Element description (25 pts) -- **a11y notes**, **keyboard interaction**
- Attribute docs (20 pts) -- purpose, constraints
- Slot docs (15 pts) -- **content guidelines**, **a11y considerations**
- CSS docs (15 pts) -- properties, parts
- Event docs (15 pts) -- triggers, detail shape
- Demos (10 pts)

### MCP: a USB port for AI {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="standard" icon="model-context-protocol" loading="eager"></rh-icon>

AI tools like Claude, Copilot, and Cursor can use **plugins**.

**MCP** (Model Context Protocol) is the standard for those plugins.
Like USB -- plug in a data source, AI can use it.

We built MCP plugins for RHDS so AI can:

- Generate valid RHDS component markup
- Check its own work against component APIs
- Query real RHDS design token values
- Follow WCAG patterns and ARIA guidance

### Architecture {data-slide-tag-name=rh-slide slot=heading}

Three layers of RHDS, each with a tool and a web standard:

| Layer      | Tool     | Standard                 |
| ---------- | -------- | ------------------------ |
| Components | CEM      | Custom Elements Manifest |
| Tokens     | Asimonim | DTCG                     |
| Modules    | Mappa    | Import Maps              |

CEM enriches component docs with real token values from Asimonim.
Mappa powers the dev server so you can preview components instantly.
