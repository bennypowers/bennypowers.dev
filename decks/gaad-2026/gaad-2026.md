## AI can hallucinate design system features

<div class="grid-2">
  <div>
    <rh-icon class="center brand-red icon size-06"
             set="standard"
             icon="code"></rh-icon>

Agents can confuse **component APIs**  
leading to **incorrect** and **inaccessible** UIs:
{.center}

- Invented attributes.
- Nonexistent slot names.
- Ignored <abbr title="accessibility">a11y</abbr> patterns.

  </div>
  <div>
    <rh-icon class="center brand-red icon size-06"
             set="ui"
             icon="ai-search"></rh-icon>

Models not trained on our **design tokens** can't use them without help
{.center}

- Hallucinated token names
- Incorrect color values
- No semantic awareness

  </div>
</div>

**Semantic awareness**: the difference between "red-30" and "brand-red",
or "blue-20" and "interactive text".

## Custom elements manifest

A **web standard format** for describing web component APIs. Attributes, slots, events, CSS properties, demos, and guidelines -- in one JSON file.

Like `package.json` for your component library. Used by IDEs, documentation tools, and AI agents to understand how components work.

</div>

---

### High performance tooling {data-slide-tag-name=rh-slide slot=heading}

## The `cem` CLI

`cem` is our tool for working with custom elements manifests.

- `cem generate` -- extract a manifest from RHDS source code
- `cem health` -- score documentation quality
- `cem serve` -- interactive component dev server
- `cem lsp` -- IDE autocomplete and validation
- `cem mcp` -- AI native access to design system elements

One source of truth powering IDE support, AI generation,
and documentation.

---

### High performance tooling {data-slide-tag-name=rh-slide slot=heading}

## The `cem health` command

Documentation quality scoring. 6 dimensions, 100 points:

- Element description (25 pts) -- **a11y notes**, **keyboard interaction**
- Attribute docs (20 pts) -- purpose, constraints
- Slot docs (15 pts) -- **content guidelines**, **a11y considerations**
- CSS docs (15 pts) -- properties, parts
- Event docs (15 pts) -- triggers, detail shape
- Demos (10 pts)

Good docs serve **cognitive a11y** for humans
AND **richer context** for AI.

### MCP: a USB port for AI {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="standard" icon="model-context-protocol"></rh-icon>

AI tools like Claude, Copilot, and Cursor can use **plugins**.

**MCP** (Model Context Protocol) is the standard for those plugins.
Like USB -- plug in a data source, AI can use it.

We built MCP plugins for RHDS so AI can:

- Generate valid RHDS component markup
- Check its own work against component APIs
- Query real RHDS design token values
- Follow WCAG patterns and ARIA guidance

### Language server {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="standard" icon="command-line"></rh-icon>

Use RHDS in your IDE with autocomplete, hover docs,
validation, and autofixes.

Works in **Twig**, Jinja2, Blade, PHP, Nunjucks,
Handlebars, Liquid, ERB, EJS --
not just HTML and TypeScript.

Meeting RHDS consumers where they are.

---

### Mappa: performance is accessibility {data-slide-tag-name=rh-slide slot=heading}

<rh-icon set="ui" icon="speedometer"></rh-icon>

RHDS components ship as unbundled ES modules.

Mappa generates **import maps** so browsers can load them
without a build step.

Smaller payloads, faster loads,
better experience on constrained devices and networks.

Performance is accessibility.

### Architecture {data-slide-tag-name=rh-slide slot=heading}

Three layers of RHDS, each with a tool and a web standard:

| Layer      | Tool     | Standard                 |
| ---------- | -------- | ------------------------ |
| Components | CEM      | Custom Elements Manifest |
| Tokens     | Asimonim | DTCG                     |
| Modules    | Mappa    | Import Maps              |

CEM enriches component docs with real token values from Asimonim.
Mappa powers the dev server so you can preview components instantly.
